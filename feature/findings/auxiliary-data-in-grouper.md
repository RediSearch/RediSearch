# Findings: Auxiliary Data in the Grouper — How Reducers Are Distributed

> Reference document: how the distribution layer rewrites reducer plans,
> where auxiliary (internal-only) data is added, transported, consumed, and omitted.
>
> Investigated: 2026-03-18  
> Cluster: 3-node OSS cluster (ports 6379, 6381, 6383)  
> Dataset: 10 JSON movie documents, 3 genres

---

## 1. Core Principle: Command Rewriting, Not Shard Awareness

The distribution mechanism works entirely through **command rewriting on the coordinator**. A shard never detects its own role or behaves differently based on context. The coordinator constructs a different command — potentially with different reducer names, different argument counts, or additional reducers — and the shard executes whatever command it receives.

All reducers live in a single flat registry. User-facing and internal reducers are peers:

```c
// src/aggregate/reducer.c:18-33
static const FuncEntry globalRegistry[] = {
  {"COUNT",              RDCRCount_New},
  {"SUM",                RDCRSum_New},
  {"TOLIST",             RDCRToList_New},
  {"MIN",                RDCRMin_New},
  {"MAX",                RDCRMax_New},
  {"AVG",                RDCRAvg_New},
  {"COUNT_DISTINCT",     RDCRCountDistinct_New},
  {"COUNT_DISTINCTISH",  RDCRCountDistinctish_New},
  {"QUANTILE",           RDCRQuantile_New},
  {"STDDEV",             RDCRStdDev_New},
  {"FIRST_VALUE",        RDCRFirstValue_New},
  {"RANDOM_SAMPLE",      RDCRRandomSample_New},
  {"HLL",                RDCRHLL_New},
  {"HLL_SUM",            RDCRHLLSum_New}
};
```

Any node can instantiate any reducer — the command determines which one runs.

The **response format is the same** regardless of which reducer runs. Both internal and external commands return rows in flat key-value structure:

```
key1  value1  key2  value2
genre "crime" <alias> <value>
```

The only difference is **what sits in the value cell** (a binary blob, a number, an array) and the alias name. The row structure is unchanged.

## 2. The Distribution Registry

A separate registry on the coordinator side maps user-facing reducer names to **distribution functions**. Each function controls how the coordinator rewrites that reducer for the shard command:

```c
// src/coord/dist_plan.cpp:340-353
static struct {
  const char *key;
  reducerDistributionFunc func;
} reducerDistributors_g[] = {
    {"COUNT",              distributeCount},
    {"SUM",                distributeSingleArgSelf},
    {"MAX",                distributeSingleArgSelf},
    {"MIN",                distributeSingleArgSelf},
    {"AVG",                distributeAvg},
    {"TOLIST",             distributeSingleArgSelf},
    {"STDDEV",             distributeStdDev},
    {"COUNT_DISTINCTISH",  distributeCountDistinctish},
    {"QUANTILE",           distributeQuantile},
    {NULL, NULL}  // sentinel
};
```

Reducers **not** in this table (`RANDOM_SAMPLE`, `FIRST_VALUE`, `COUNT_DISTINCT`) are not distributed — the GROUPBY is not included in the shard command. Shards return raw rows and the coordinator runs the entire GROUPBY locally.

## 3. How the Rewriting Works — `distributeGroupStep`

When the coordinator processes a query, `AGGPLN_Distribute` calls `distributeGroupStep` to split a GROUPBY into two plans:

- **Remote plan** — serialized into `_FT.AGGREGATE` and sent to each shard
- **Local plan** — runs on the coordinator against the shard responses

```c
// src/coord/dist_plan.cpp:108-136
static void distributeGroupStep(AGGPlan *origPlan, AGGPlan *remote, PLN_BaseStep *step,
                                PLN_DistributeStep *dstp, QueryError *status) {
  PLN_GroupStep *gr = (PLN_GroupStep *)step;
  PLN_GroupStep *grLocal = PLNGroupStep_New(...);
  PLN_GroupStep *grRemote = PLNGroupStep_New(...);

  // ...

  for (size_t ii = 0; ii < nreducers; ii++) {
    rdctx.srcReducer = gr->reducers + ii;
    reducerDistributionFunc fn = getDistributionFunc(gr->reducers[ii].name);
    if (!fn || fn(&rdctx, status) != REDISMODULE_OK) {
      goto cleanup;
    }
  }
  AGPLN_AddStep(remote, &grRemote->base);
}
```

The rewriting happens **per-reducer** — in a single GROUPBY with multiple reducers, each one is independently rewritten by its own distribution function. This is the granularity of the mechanism.

For each reducer, the distribution function calls:
- `addRemote` — adds a reducer to the shard command (potentially with a different name)
- `addLocal` — adds a reducer to the coordinator plan (reads from the shard's output alias)

### Auto-generated aliases

When `addRemote` creates a reducer in the shard command, the alias is auto-generated by `getReducerAlias`:

```c
// src/aggregate/aggregate_request.c:764-783
static char *getReducerAlias(PLN_GroupStep *g, const char *func, const ArgsCursor *args) {
  sds out = sdsnew("__generated_alias");
  out = sdscat(out, func);
  // Concatenates all args (stripping leading '@')
  // ...
}
```

This produces aliases like `__generated_aliashlllength` (for `REDUCE HLL 1 length`).

The returned alias is then passed into `addLocal` so the coordinator's reducer reads from the column the shard produced.

### Deduplication of remote reducers

`addRemote` checks if the same reducer already exists in the remote plan before adding it. This handles cases like `COUNT` + `AVG` both needing a `COUNT` in the shard command:

```c
// src/coord/dist_plan.cpp:83-97
bool addRemote(const char *name, const char **alias, QueryError *status, T... uargs) {
  // ...
  auto existing = PLNGroupStep_FindReducer(remoteGroup, name, &tmp);
  if (existing) {
    if (alias) *alias = existing->alias;
    return true;  // Reuse existing
  } else {
    return add(remoteGroup, name, alias, status, cargs);
  }
}
```

## 4. Complete Distribution Map

Every distributed reducer and what the coordinator rewrites it to:

| User writes | Shard command gets | Coordinator plan gets | Value type change | Notes |
|---|---|---|---|---|
| `COUNT` | `COUNT` | `SUM` | number → number | Partial counts summed |
| `SUM` | `SUM` | `SUM` | number → number | Partial sums summed |
| `MAX` | `MAX` | `MAX` | number → number | Max of maxes |
| `MIN` | `MIN` | `MIN` | number → number | Min of mins |
| `TOLIST` | `TOLIST` | `TOLIST` | array → array | Shard arrays unpacked, deduplicated |
| `AVG` | `COUNT` + `SUM` | `SUM` + `SUM` + `APPLY` | number, number → number | Intermediate columns hidden via `isHidden` |
| `COUNT_DISTINCTISH` | `HLL` | `HLL_SUM` | binary blob → number | Most dramatic type change |
| `QUANTILE` | `RANDOM_SAMPLE` (size 500) | `QUANTILE` | array → number | Samples merged, statistic computed |
| `STDDEV` | `RANDOM_SAMPLE` (size 500) | `STDDEV` | array → number | Samples merged, statistic computed |

Reducers without a distribution entry (`RANDOM_SAMPLE`, `FIRST_VALUE`, `COUNT_DISTINCT`) — the GROUPBY is not pushed to shards. Shards return raw rows, coordinator groups locally.

## 5. Distribution Function Code

### Same Name Both Sides — `distributeSingleArgSelf`

Used by: `SUM`, `MAX`, `MIN`, `TOLIST`

```c
// src/coord/dist_plan.cpp:226-239
static int distributeSingleArgSelf(ReducerDistCtx *rdctx, QueryError *status) {
  PLN_Reducer *src = rdctx->srcReducer;
  CHECK_ARG_COUNT(1);
  const char *alias;
  if (!rdctx->addRemote(src->name, &alias, status, "1", rdctx->srcarg(0))) {
    return REDISMODULE_ERR;
  }
  if (!rdctx->addLocal(src->name, status, "1", alias, "AS", src->alias)) {
    return REDISMODULE_ERR;
  }
  return REDISMODULE_OK;
}
```

### Multiple Reducers + Pipeline Steps — `distributeAvg`

```c
// src/coord/dist_plan.cpp:294-334 (abbreviated)
static int distributeAvg(ReducerDistCtx *rdctx, QueryError *status) {
  // Shard command gets two reducers
  rdctx->addRemote("COUNT", &remoteCountAlias, status, "0");
  rdctx->addRemote("SUM", &remoteSumAlias, status, "1", rdctx->srcarg(0));

  // Coordinator plan: SUM the counts, SUM the sums (both hidden)
  rdctx->add(local, "SUM", &localCountSumAlias, status, "1", remoteCountAlias);
  array_tail(rdctx->localGroup->reducers).isHidden = 1;
  rdctx->add(local, "SUM", &localSumSumAlias, status, "1", remoteSumAlias);
  array_tail(rdctx->localGroup->reducers).isHidden = 1;

  // Then APPLY (@sum / @count)
  PLN_MapFilterStep *applyStep = ...;
  AGPLN_AddAfter(rdctx->localPlan, rdctx->currentLocal, &applyStep->base);
}
```

### No Distribution (Fallback)

Reducers not in `reducerDistributors_g[]` → `getDistributionFunc` returns `NULL` → `distributeGroupStep` fails at line 133 → cleanup → the GROUPBY is not included in the shard command. Shards return raw (ungrouped) rows.

## 6. Where Auxiliary Columns Are Omitted — `isHidden` + `RLOOKUP_F_HIDDEN`

When the coordinator needs intermediate columns that the user should not see (like the raw `SUM` and `COUNT` columns in AVG distribution), the distribution function marks those reducers as hidden:

```c
// src/coord/dist_plan.cpp:316,320
array_tail(rdctx->localGroup->reducers).isHidden = 1; // Don't show this in the output
```

During pipeline construction, `isHidden` maps to the `RLOOKUP_F_HIDDEN` flag on the output key:

```c
// src/pipeline/pipeline_construction.c:63-65
uint32_t flags = pr->isHidden ? RLOOKUP_F_HIDDEN : RLOOKUP_F_NOFLAGS;
RLookupKey *dstkey = RLookup_GetKey_Write(&gstp->lookup, pr->alias, flags);
```

During result serialization, keys with `RLOOKUP_F_HIDDEN` are excluded:

```c
// src/aggregate/aggregate_exec.c:242
uint32_t excludeFlags = RLOOKUP_F_HIDDEN;
```

Currently, `isHidden` is only used in `distributeAvg`. Other patterns (like HLL) don't need it because the coordinator-side reducer writes directly to the user-facing alias.

## 7. Example: COUNT_DISTINCTISH → HLL / HLL_SUM

This section illustrates the general mechanism with the most dramatic case — where the shard command gets a completely different reducer that produces a binary blob, and the coordinator consumes and transforms it into a number.

### 7a. Three independent reducers in the flat registry

`HLL`, `HLL_SUM`, and `COUNT_DISTINCTISH` are three separate entries in `globalRegistry`. They share internal implementation code, but from the command's perspective they are independent reducers:

- `COUNT_DISTINCTISH` — accumulates into HLL registers, `Finalize` calls `hll_count()` → returns a number
- `HLL` — accumulates into HLL registers, `Finalize` serializes the registers → returns a binary string
- `HLL_SUM` — `Add` deserializes binary strings and merges registers, `Finalize` calls `hll_count()` → returns a number

They share the internal constructor `newHllCommon`, where `isRaw` is a **construction-time parameter** passed by the factory function:

```c
// src/aggregate/reducers/count_distinct.c:137-164
static Reducer *newHllCommon(const ReducerOptions *options, int isRaw) {
  Reducer *r = rm_calloc(1, sizeof(*r));
  // ...
  r->Add = distinctishAdd;  // Same accumulation logic

  if (isRaw) {
    r->reducerId = REDUCER_T_HLL;
    r->Finalize = hllFinalize;        // Serialize registers → binary string
  } else {
    r->reducerId = REDUCER_T_DISTINCTISH;
    r->Finalize = distinctishFinalize; // hll_count() → number
  }
  return r;
}

Reducer *RDCRHLL_New(const ReducerOptions *options) {
  return newHllCommon(options, 1);  // Always isRaw=1
}
```

`RDCRHLL_New` always passes `isRaw=1`. `RDCRCountDistinctish_New` always passes `isRaw=0`. Which runs is determined entirely by which name appears in the command.

### 7b. Shard output — `hllFinalize`

```c
// src/aggregate/reducers/count_distinct.c:124-135
static RSValue *hllFinalize(Reducer *parent, void *ctx) {
  distinctishCounter *ctr = ctx;
  HLLSerializedHeader hdr = {.flags = 0, .bits = ctr->hll.bits};
  char *str = rm_malloc(sizeof(hdr) + ctr->hll.size);
  memcpy(str, &hdr, sizeof(hdr));
  memcpy(str + sizeof(hdr), ctr->hll.registers, ctr->hll.size);
  return RSValue_NewString(str, sizeof(hdr) + ctr->hll.size);
}
```

Output: `RSValue` string — 5-byte header + 256 register bytes = 261 bytes.

### 7c. Coordinator consumption — `hllsumAdd`

```c
// src/aggregate/reducers/count_distinct.c:168-216
static int hllsumAdd(Reducer *r, void *ctx, const RLookupRow *srcrow) {
  hllSumCtx *ctr = ctx;
  const RSValue *val = RLookupRow_Get(r->srckey, srcrow);

  if (val == NULL || !RSValue_IsString(val)) return 0;

  size_t len;
  const char *buf = RSValue_StringPtrLen(val, &len);
  const HLLSerializedHeader *hdr = (const void *)buf;
  const char *registers = buf + sizeof(*hdr);

  // Validate header and register size ...

  if (ctr->bits) {
    struct HLL tmphll = {.bits = hdr->bits, .size = 1 << hdr->bits, .registers = (uint8_t *)registers};
    hll_merge(ctr, &tmphll);
  } else {
    hll_load(ctr, registers, len - sizeof(*hdr));
  }
  return 1;
}
```

### 7d. Coordinator output — `hllsumFinalize`

```c
// src/aggregate/reducers/count_distinct.c:218-221
static RSValue *hllsumFinalize(Reducer *parent, void *ctx) {
  hllSumCtx *ctr = ctx;
  return RSValue_NewNumber(ctr->bits ? (uint64_t)hll_count(ctr) : 0);
}
```

Output: `RSValue` number — the estimated distinct count. This is what the user sees.

### 7e. How the alias threads through

The alias returned by `addRemote("HLL", &alias, ...)` is the auto-generated `__generated_aliashlllength`. This alias is passed as the input argument to `addLocal("HLL_SUM", ..., "1", alias, ...)`. So `HLL_SUM`'s `srckey` resolves to the column that contains the binary blob.

## 8. Captured Terminal Output

### 8a. External command → MONITOR → Internal command

**User sends** (via Python, RESP3, to coordinator port 6379):
```
FT.AGGREGATE idx:movies "*"
  LOAD 2 @length @genre
  GROUPBY 1 @genre
    REDUCE COUNT_DISTINCTISH 1 @length AS distinct_lengths
```

**Coordinator MONITOR** (port 6379) shows incoming + outgoing:
```
"FT.AGGREGATE" "idx:movies" "*" "LOAD" "2" "@length" "@genre"
  "GROUPBY" "1" "@genre" "REDUCE" "COUNT_DISTINCTISH" "1" "@length" "AS" "distinct_lengths"

"_FT.AGGREGATE" "idx:movies" "*" "WITHCURSOR" "_NUM_SSTRING"
  "_INDEX_PREFIXES" "1" "movie:" "_SLOTS_INFO" "\x01\x00\x00\x00\x00\x00U\x15"
  "LOAD" "2" "@length" "@genre"
  "GROUPBY" "1" "@genre"
    "REDUCE" "HLL" "1" "length" "AS" "__generated_aliashlllength"
  "_COORD_DISPATCH_TIME" "198001"
```

Key observations:
- The reducer name changed: `COUNT_DISTINCTISH` → `HLL`
- The alias changed: `AS distinct_lengths` → `AS __generated_aliashlllength` (auto-generated)
- The field arg lost its prefix: `@length` → `length` (stripped by `stripAtPrefix`)
- Internal prefixes added: `WITHCURSOR`, `_NUM_SSTRING`, `_INDEX_PREFIXES`, `_SLOTS_INFO`, `_COORD_DISPATCH_TIME`

**Shard MONITOR** (port 6383) confirms the same command arriving:
```
"_FT.AGGREGATE" "idx:movies" "*" "WITHCURSOR" "_NUM_SSTRING"
  "_INDEX_PREFIXES" "1" "movie:" "_SLOTS_INFO" "\x01\x00\x00\x00\xac*\xff?"
  "LOAD" "2" "@length" "@genre"
  "GROUPBY" "1" "@genre"
    "REDUCE" "HLL" "1" "length" "AS" "__generated_aliashlllength"
  "_COORD_DISPATCH_TIME" "173001"
```

Note: `_SLOTS_INFO` differs per shard — each receives its own slot range.

### 8b. Shard response — same row format, different value type

Directly executing the internal command on the coordinator node (port 6379) as a shard with `_SLOTS_INFO "\x01\x00\x00\x00\x00\x00U\x15"` (slots 0–5461):

```
1) 1) (integer) 3
   2) 1) "genre"
      2) "crime"
      3) "__generated_aliashlllength"
      4) "\x00\x00\x00\x00\b\x00\x00\x00\x00\x00...
           \x01\x00...\x02\x00..."          ← 261 bytes: HLL binary blob
   3) 1) "genre"
      2) "scifi"
      3) "__generated_aliashlllength"
      4) "\x00\x00\x00\x00\b\x00\x00\x00..."  ← HLL binary blob
   4) 1) "genre"
      2) "horror"
      3) "__generated_aliashlllength"
      4) "\x00\x00\x00\x00\b\x00\x00\x00..."  ← HLL binary blob
2) (integer) 0
```

The row format is identical to any other response: `genre <value> __generated_aliashlllength <value>`. The difference is only in what sits in the value cell — a 261-byte binary blob instead of a number or array.

### 8c. Prerequisites for executing `_FT.AGGREGATE` directly

1. **`DEBUG MARK-INTERNAL-CLIENT`** — marks the connection as internal (otherwise `_FT.AGGREGATE` is rejected as unknown command)
2. **`_SLOTS_INFO <binary>`** — required binary argument specifying which hash slots to process. Format: `4-byte int32 (range count) + pairs of uint16 (start, end)`
3. **`HELLO 3`** — if RESP3 is needed, must be sent before the command

Without `MARK-INTERNAL-CLIENT`:
```
(error) ERR unknown command '_FT.AGGREGATE', with args beginning with: ...
```

Without `_SLOTS_INFO`:
```
(error) SEARCH_FIELD_MISSING_REQ Internal query missing slots specification
```
