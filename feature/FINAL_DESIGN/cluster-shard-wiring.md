# Cluster Shard Wiring: Enhanced TOLIST Wire Format

## Overview

This document describes the internal wire format between shard and coordinator for the enhanced TOLIST reducer, and how the coordinator's reader function consumes it.

---

## Key Insight: Reducer Column Isolation

Each reducer reads **only its own column** in the coordinator's `RLookupRow`. The TOLIST reducer on the coordinator reads from the alias that the shard's TOLIST wrote to (auto-generated by `distributeSingleArgSelf` via `getReducerAlias`). No other reducer touches that column, and no other column is affected by what TOLIST puts in it.

This means the wire format between shard TOLIST and coordinator TOLIST is a **private contract**. Changing it is transparent to every other reducer and to the user.

### The Reader Function

The function that reads shard output on the coordinator is **`tolistAdd`** (`src/aggregate/reducers/to_list.c:38`). It's the same function used on both shard and coordinator today. On the shard, `srckey` resolves to a field in the pipeline row (scalar). On the coordinator, `srckey` resolves to the shard's auto-generated alias, which contains the shard's TOLIST output (an array).

The current `tolistAdd` has two branches:
- **Non-array** (shard side): `dictAdd(values, v, NULL)` — insert scalar directly.
- **Array** (coordinator side): iterate elements, `dictAdd` each one — unpack shard output.

---

## Wire Format: Uniform Array-of-Tuples

### Format

The shard TOLIST `Finalize()` **always** produces an array of tuples. Every entry is wrapped in an array:

**With SORTBY (N sort keys):**
```
[
  [value_1, sortkey1_1, sortkey2_1],
  [value_2, sortkey1_2, sortkey2_2],
  [value_3, sortkey1_3, sortkey2_3]
]
```

**Without SORTBY:**
```
[
  [value_1],
  [value_2],
  [value_3]
]
```

### Rules

- Element 0 of each tuple is **always the value** (the payload — field value for `@field`, document map for `*`).
- Elements 1..N are sort key values (if SORTBY is present).
- Tuple length = 1 + number of sort keys.
- This format applies to **all** TOLIST — including legacy `TOLIST 1 @field`. No split between "legacy path" and "enhanced path" on the wire.

### Why Uniform

- **One shard `Finalize()`** — always wraps entries in tuples.
- **One coordinator `Add()`** — always unwraps element 0 as the value. If tuple length > 1, extracts sort keys for heap-based merge. If tuple length == 1, just the value (dedup dict, no heap).
- **Zero branching** on "is this legacy or enhanced" at the wire format level. The only variable is tuple length.
- **Rolling upgrade compatibility is not a concern** — module upgrades are atomic across all nodes.

### User-Facing Output Is Unchanged

The wrapping is purely internal transport. The coordinator's `Finalize()` outputs bare values (or maps for `TOLIST *`). The user never sees the tuples — they see the same response format as today.

---

## Coordinator `Add()` Behavior

The enhanced `tolistAdd` (or its replacement) on the coordinator:

1. Reads the alias column → gets `RSValue_Array` (the shard's output).
2. Iterates the outer array. Each element is a **tuple** (itself an `RSValue_Array`).
3. For each tuple:
   - **Unwraps element 0** → the value (payload).
   - **Extracts elements 1..N** → sort key values (if present).
4. If SORTBY is present:
   - Inserts `{ payload, sortkeys[] }` into a bounded heap (same heap infrastructure as the shard-side in-group sorter).
   - Heap merges entries across shards, re-sorts, applies cross-shard dedup (if not ALLOWDUPS).
5. If SORTBY is absent:
   - Inserts the unwrapped value into the dedup dict (same as today's behavior, just with an unwrap step).
6. On `Finalize()`: extracts from heap (or dict), outputs bare values only — sort keys stripped.

---

## How N Is Known

Both shard and coordinator parse the same forwarded SORTBY clause. The number of sort keys N is implicit — derived from the reducer's own construction args. No out-of-band metadata needed. Both sides agree on N by construction.

---

## Distribution Function

`distributeSingleArgSelf` currently hardcodes `CHECK_ARG_COUNT(1)` and forwards only the field name. For enhanced TOLIST, a new distribution function (`distributeToList`) forwards the full arg set: `@field` (or `*`), `ALLOWDUPS`, `SORTBY ...`, `LIMIT ...`.

The existing serialization infrastructure already handles arbitrary reducer args — no changes needed there.

---

## Backward Compatibility

| Scenario | Shard Output | Coordinator Behavior |
|---|---|---|
| Legacy `TOLIST 1 @field` | `[[val1], [val2], [val3]]` | Unwrap element 0 from each tuple, dedup dict |
| Enhanced with SORTBY | `[[val1, sk1, sk2], [val2, sk1, sk2], ...]` | Unwrap element 0 + extract sort keys, heap merge |
| Enhanced without SORTBY | `[[val1], [val2], [val3]]` | Same as legacy — unwrap, dedup dict |

The user-facing response is identical in all cases. The wire format change is internal only.

## Open Questions

None — the design is settled. The uniform tuple format eliminates all branching concerns at the wire level.
