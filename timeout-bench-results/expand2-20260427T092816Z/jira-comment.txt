Benchmark completed locally for the ON_TIMEOUT=FAIL strictness comparison.

TL;DR

The new mechanism on master is generally stricter than the old v8.6.0 in-pipeline timeout mechanism.

Across the full completed benchmark set, in cells where both versions timed out 100% of the time:

v8.6.0 mean overshoot: 269.3 ms
master mean overshoot: 198.9 ms
delta: -70.4 ms, about 26% less overshoot
cell count: 34 apples-to-apples cells
cell direction: master better in 26 cells, worse in 8

Using a looser threshold of cells where both versions timed out at least 50% of the time gives the same result:

v8.6.0 mean overshoot: 262.3 ms
master mean overshoot: 193.4 ms
delta: about 26% less overshoot

Main conclusion: for FAIL, the new mechanism is not too strict. It is generally closer to the requested timeout, especially for FT.AGGREGATE. FT.SEARCH is mostly similar with small wins for master on short-timeout cluster cases. FT.HYBRID has one caveat at very tight timeouts, described below.

Workload notes

FT.AGGREGATE:
Strongest improvement. master is consistently closer to the requested timeout in cells where timeout enforcement actually triggers, especially at 500K/1M docs and on wider clusters.

FT.SEARCH:
Mostly similar, with small wins for master on short-timeout cluster cases, for example cluster7 / 1M / TIMEOUT 100 ms.

FT.HYBRID:
This is the one caveat. In the earlier 1M runs:
* standalone / HYBRID / TIMEOUT 50 ms:
  v8.6.0 mean overshoot was +0.4 ms
  master mean overshoot was +50.8 ms
* cluster3 / HYBRID / TIMEOUT 50 ms:
  v8.6.0 mean overshoot was +0.37 ms
  master mean overshoot was +41.5 ms

The later cluster5 and cluster7 HYBRID 1M rows mostly did not hit the timeout budget because natural latency was under or near 50 ms.

server_mean_ms note

The new server_mean_ms column is useful, but it should not replace client-observed wall time for this comparison.

On v8.6.0, server_mean_ms usually tracks observed_ms closely. On master, timeout-heavy worker-thread paths often report near-zero server_mean_ms even when the client waits tens or hundreds of milliseconds. This means INFO commandstats under-reports work attributable to timed-out queries on the new mechanism.

Examples from the latest run:

* cluster3 / FT.SEARCH / 500K / TIMEOUT 50 ms
  observed old/new: 86.4 / 63.3 ms
  server_mean old/new: 86.1 / 0.12 ms

* cluster3 / FT.AGGREGATE / 500K / TIMEOUT 500 ms
  observed old/new: 553.6 / 518.8 ms
  server_mean old/new: 553.3 / 0.21 ms

* cluster5 / FT.AGGREGATE / 1M / TIMEOUT 100 ms
  observed old/new: 116.4 / 105.1 ms
  server_mean old/new: 116.1 / 0.14 ms

* cluster7 / FT.AGGREGATE / 1M / TIMEOUT 50 ms
  observed old/new: 60.2 / 54.8 ms
  server_mean old/new: 59.9 / 0.13 ms

* standalone / FT.AGGREGATE / 500K / TIMEOUT 100 ms
  observed old/new: 1057.0 / 1023.8 ms
  server_mean old/new: 1056.7 / 0.19 ms

So the main metric for MOD-14527 should remain client-observed wall time and overshoot. server_mean_ms is a useful operator-facing caveat: timed-out queries on master may not be represented accurately in commandstats.

Caveats

* Long-timeout AGGREGATE cells sometimes run to natural latency without firing the timeout. Those rows are useful behaviorally, but they should not be treated as apples-to-apples timeout overshoot unless timeout_pct is high on both versions.
* HYBRID exact-deadline strictness at very tight timeouts should be tracked separately if it matters. It is the only clear opposite-direction result.
* This is still a local single-host benchmark with N=50. It answers the yes/no and rough-magnitude question, not an absolute SLA question.
